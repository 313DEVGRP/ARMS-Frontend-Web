Chance in Crisis ( 배경 )

비지니스 범주 확대 실패 (Scope)
비지니스를 확대하기 위한 전략으로 계열사 및 분사를 하게됩니다. 비지니스 범주의 실패는 자회사 및 계열사의 위기가 공동체에 영향을 주게되는 현상에 직면합니다. 일관된 비지니스 범주 관리는 공동체에서 시작되어 지속적인 Align을 맞추는데에 그 해법이 있습니다.
비지니스 효율성 관리 실패 (Resource)
비지니스의 효율성은 그 중요성이 지나침이 없습니다. 비효율적인 자원 배분과, 그 관리의 효율성은 곧바로 비지니스의 경쟁력에 연계됩니다. 일관된 비지니스 효율성 관리는 인적 자원에서부터 시작되고 마무리됨을 기억해야 합니다.
비지니스 적시성 실패 (Time)
비지니스는 적절한 시기와 함께 성장하고 발전하는 경향이 있습니다. 우리는 이 비지니스 적시성이 성공과 실패를 가름함을 잘 알고 있습니다. 실제 비지니스의 최전선에 있는 제품(서비스)가 적시에 배포되고 계획할 수 있는지는 매우 중요한 지표입니다.

제품(서비스)가 되고 있습니까? ( Time )

제품(서비스)의 되고 있습니까? ( Scope )

제품(서비스)의 기능 가 얼마인지 아십니까? ( Cost )

Effect ( 효과 )

비지니스 가시성 증대 (Time)
제품(서비스)의 요구사항을 ALM과 연결하여, 요구사항 기반의 정확한 진척도를 확인 할 수 있습니다. 정량적인 데이터를 기반으로 비지니스의 가시성을 제공하여, 결정의 근거를 제공합니다.
비지니스 이해도 증대 (Scope)
제품(서비스)의 기획 부터, 요구사항 ~ TASK ( Issue ) 에 참여된 모든 관계자들의 소통 채널을 제공하고, 업무 시스템에 연결하여, 비지니스 목표에 대한 이해도를 공유할 수 있습니다.
비지니스 효율성 증대 (Resource)
제품(서비스) 기반의 요구사항을 ALM TASK(Issue) 로 전달 후 관련 업무 처리에 대한 담당자의 효율적인 업무 처리를 위한 지원을 제공하며, 통계화 하여 정확한 성과 지표를 제공합니다.

Features ( 기능 )

모든 회사는 제품 혹은 서비스 를 만들고 있습니다.


제품(서비스) 관점으로 관리할 수 있고, 기획서를 온라인으로 관리할 수 있도록 기능을 제공하고 있습니다.

모든 제품(서비스) 에는 버전이 존재합니다.


제품(서비스) 의 버전은 출시일이 정해져있고, 출시할 기능을 정의합니다. A-RMS는 버전을 관리 할 수 있습니다.

모든 회사는 ALM ( Jira, Redmine, GitLab ... etc ) 을 사용하고 있습니다.


A-RMS 는 ALM Server 및 Cloud 를 관리할 수 있으며, 필요에 따라 동시 사용이 가능합니다.

제품(서비스) 의 버전과 ALM Server를 연결하여 관리 할 수 있습니다.


제품(서비스) 의 버전이 없다면, 초기 등록된 Base Version 을 사용하여 연결하면 됩니다.

제품(서비스) 의 버전에 요구사항을 등록하과 관리 할 수 있습니다.


제품(서비스) 관점으로 요구사항을 Tree 형태로 관리하는 기능을 제공합니다 ( 생성, 이동, 삭제 등 )

제품(서비스) 에 등록된 요구사항 이슈와 연결된 이슈까지 수집하여,


제품(서비스) 의 요구사항을 관점으로 BA 통계를 제공합니다 ( Time, Scope, Resource-Cost )
